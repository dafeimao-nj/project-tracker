## 一、项目背景

### 1.1 需求来源
用户是数科公司整体业务运营负责人，负责AI能力的落地推进，需要跟进大量工作事项，每个事项需要专人跟进，需要记录最新动态、进展、预计完成时间等。

### 1.2 核心场景
- 跟进多个AI能力落地项目
- 每个项目有专门负责人
- 需要记录进展动态
- 需要预计完成时间和实际进度

## 二、需求梳理过程

### 2.1 初始需求确认

**用户提问**：
> 我是数科公司整体业务运营的负责人，现在针对AI能力的落地作推进。跟进AI能力的落地、跟进、效果评估等多个类型的工作。我需要做一个系统来管理我的工作事项。

**AI引导确认**：
1. 数据存储方案 → 本地Web应用
2. 多用户需求 → 单人使用（后续可扩展）
3. 手机访问需求 → 需要手机端访问
4. 负责人范围 → 固定团队成员
5. 跟进记录字段 → 时间、人、内容

### 2.2 字段设计

**基础字段**：
- 事项名称（下拉选择：风控模型的搭建 / 智能客服 / 智能巡检）

- 负责人（下拉选择：黄京 / 夏韵 / 孙培 / 赵园园 / 姜玉萍）
- 优先级（低 / 中 / 高 / 紧急）
- 状态（未启动 / 进行中 / 遇阻 / 待验收 / 已完成 / 已取消）
- 预计完成时间
- 实际完成时间
- 创建时间

**跟进记录字段**：
- 跟进时间（自动生成）
- 跟进人（自动获取）
- 跟进内容（文本）

## 三、技术方案演进

### 3.1 方案一：本地全栈版本（已弃用）

**技术栈**：
- 后端：Python FastAPI + SQLite
- 前端：原生 HTML/CSS/JS
- 端口：8000

**问题**：
- 需要启动后端服务
- 手机只能局域网访问（192.168.1.21:8000）
- 无法在非局域网环境使用

**文件位置**：`~/.qclaw/workspace/项目跟进系统/`

### 3.2 方案二：纯前端版本（过渡方案）

**技术栈**：
- 单个HTML文件
- localStorage存储数据
- 无后端依赖

**问题**：
- iOS Safari限制，无法直接打开本地HTML文件
- 需要通过iCloud/微信等方式间接访问
- 用户体验差

**文件位置**：
- `~/.qclaw/workspace/index.html`
- `~/Desktop/项目跟进系统.html`

### 3.3 方案三：GitHub Pages部署（最终方案）

**技术栈**：
- 单个HTML文件
- localStorage存储数据
- GitHub Pages托管

**优势**：
- ✅ 永久免费
- ✅ 公网访问，任何设备都能打开
- ✅ 可分享给同事
- ✅ 无需服务器

**访问地址**：
```
https://dafeimao-nj.github.io/project-tracker/project-tracker.html
```

**GitHub仓库**：
